# 전세사기 예방 앱 구축 가능성 분석 및 기획서

현재 보유하고 있는 대한민국 법률 및 행정규칙 데이터베이스(`legalize-kr`, `admrule-kr`)를 기반으로 전세사기 예방 앱을 만들 수 있는지 분석하고, 필요한 구성 요소들을 정리한 기획서입니다.

---

## 1. 현재 코드 및 데이터로 가능한 범위

보유 중인 데이터는 대한민국의 **모든 법률 및 행정규칙의 텍스트 데이터베이스**입니다. 이를 활용하면 다음과 같은 기능을 구현할 수 있습니다.

### 💡 RAG(검색 증강 생성) 기반 법률 상담 챗봇
* **설명**: 사용자가 "전세 계약 후 확정일자를 받았는데 임대인이 당일에 저당권을 설정하면 어떻게 되나요?"와 같은 질문을 했을 때, `주택임대차보호법` 및 관련 판례/행정규칙 데이터를 실시간으로 참조하여 정확한 법률적 조언을 제공하는 AI 챗봇을 만들 수 있습니다.
* **핵심 이점**: 일반적인 LLM이 겪는 환각 현상(Hallucination) 없이, 실제 법령에 근거한 신뢰성 높은 답변을 생성할 수 있습니다.

### 📝 전세 계약 맞춤형 특약 사항 추천
* **설명**: 임차인에게 불리한 조항을 걸러내고 법적으로 자신을 보호할 수 있는 필수 특약 사항(예: 국세/지방세 완납 증명 제출 의무, 대항력 확보 전 담보권 설정 금지 등)을 법령에 기반하여 자동으로 추천해 줍니다.

### 📘 단계별 법률 가이드 및 체크리스트
* **설명**: 전세 계약 전, 계약 당일, 계약 후 각 단계별로 확인해야 하는 법적 의무 사항과 대항력 발생 요건 등을 시각적으로 정리한 체크리스트를 제공합니다.

## 2. 실질적인 전세사기 방지를 위해 '추가로' 필요한 것들

단순히 법률 조문을 아는 것만으로는 실제 교묘하게 발생하는 전세사기를 완벽히 막을 수 없습니다. 전세사기는 대부분 **매물의 실시간 권리관계 유출, 시세 조작, 임대인의 신용 문제**에서 발생하기 때문에, 이를 방지하려면 외부 데이터 연동이 필수적입니다.

### 🔍 1) 외부 데이터 및 API 연동
| 필요 데이터 | 제공 기관 / API 예시 | 앱에서의 활용 목적 |
| :--- | :--- | :--- |
| **부동산 등기부등본** | 대법원 인터넷등기소 (민간 스크래핑/조회 API 활용) | 근저당권 설정 여부, 선순위 채권 금액 확인, 압류/가압류 등 권리관계 실시간 분석 |
| **건축물대장** | 정부24 (공공데이터포털 API) | 위반건축물 여부 확인 (위반건축물은 전세대출 및 보증보험 가입이 불가능함) |
| **주택 실거래가 및 시세** | 국토교통부 실거래가 API, KB부동산 | 해당 매물의 전세가율(전세가/매매가)을 계산하여 '깡통전세' 위험도 측정 |
| **임대인 세금 체납 정보** | 국세청 / 행정안전부 API | 임대인의 미납 국세/지방세 여부 확인 (임대인 체납 세금은 보증금보다 우선 변제되므로 필수 확인 필요) |
| **악성 임대인 명단** | HUG(주택도시보증공사) 안심전세 API | 과거 보증금을 돌려주지 않은 악성 임대인(블랙리스트) 여부 조회 |

### 🛠️ 2) 추가 개발이 필요한 기술 스택
1. **백엔드 서버 (API 중계 및 분석 엔진)**
   * 등기부등본 및 정부 공공 API에서 데이터를 받아와 위험도 점수(Risk Score)를 계산하는 백엔드 로직이 필요합니다. (Node.js, FastAPI 등)
2. **벡터 데이터베이스 (Vector DB)**
   * 법률 마크다운 문서들(`legalize-kr`, `admrule-kr`)을 임베딩하여 빠르고 정확하게 의미론적 검색을 수행할 수 있도록 지원하는 DB(예: Chroma, Pinecone, pgvector 등)가 필요합니다.
3. **사용자 인터페이스 (Frontend)**
   * 모바일 앱(Flutter, React Native 등) 또는 반응형 웹앱(Next.js, Tailwind CSS 등)을 통해 사용자가 주소만 입력하면 직관적으로 위험도를 파악할 수 있는 화면이 필요합니다.

## 3. 추천 앱 아키텍처 및 핵심 기능안

### 1단계: 주소 입력 및 매물 진단 (가장 핵심)
1. 사용자가 매물 주소와 전세 보증금 액수를 입력합니다.
2. 앱이 API를 통해 **등기부등본**과 **건축물대장**을 조회합니다.
3. 등기부등본 상의 채권최고액(융자) + 선순위 보증금 + 사용자의 보증금 합산액이 주변 시세의 몇 %인지 분석합니다. (80% 이상일 경우 깡통전세 경고)
4. 미납 국세 여부 및 위반건축물 여부를 종합하여 **"안전성 등급(정상/주의/위험)"**을 도출합니다.

### 2단계: 법률 챗봇 및 특약 생성 (보유 데이터 활용)
1. 진단 결과에 따라 발생할 수 있는 법적 위험(예: 선순위 근저당으로 인한 경매 시 보증금 변제 순위)을 설명합니다.
2. 계약서 작성 시 임대인에게 요구해야 할 **특약 사항 마크다운 문서**를 자동 생성하여 복사할 수 있도록 제공합니다.
3. 사용자가 전세 계약과 관련하여 궁금한 점을 질문하면 법령 기반으로 답변하는 챗봇 서비스를 제공합니다.

## 4. 향후 로드맵 제안

1. **Phase 1: 지식 베이스 구축 (현재 데이터 활용)**
   * 현재 보유한 대한민국 법률 데이터를 Vector DB에 적재하고, LLM과 연동하여 **전세사기 전문 법률 상담 챗봇**을 먼저 구현합니다.
2. **Phase 2: 공공데이터 연동**
   * 국토교통부 실거래가 및 건축물대장 API를 연동하여 시세 대비 전세가율을 계산하는 **깡통전세 계산기**를 구축합니다.
3. **Phase 3: 권리 분석 자동화**
   * 등기부등본 PDF/텍스트 파싱 API를 도입하여 등기부등본 상의 위험 요소를 자동으로 추출하는 **권리 분석 엔진**을 적용합니다.

## 5. 2차 기능 확장 계획 (사용자 피드백 기반)

사용자가 요청한 세 가지 추가 핵심 기능을 바탕으로 한 기술 사양 및 구현 시나리오입니다.

### 📄 기능 1: 임대차 계약서 스캔 및 위험 감지 (AI OCR & 비전 분석)
* **목적**: 사용자가 직접 찍은 계약서 사진(JPG)이나 PDF를 업로드하면, 독소 조항이나 임차인에게 불리한 누락 사항을 자동 분석.
* **구현 방안**:
  1. **프론트엔드**: 드래그 앤 드롭 파일 업로드 컴포넌트 추가 (`src/components/ContractUploader.tsx`).
  2. **API 연동**: Next.js API Routes에서 OpenAI의 **GPT-4o (Vision 지원 모델)** API를 호출.
  3. **비전 프롬프팅**: GPT-4o에 임대차계약서 이미지를 입력값으로 전달하고, "이 계약서 내에서 임차인에게 불리하게 작용할 수 있는 불공정 특약이나 독소 조항이 있는지 찾아내고, 보증금 보호를 위해 추가해야 할 특약을 항목별로 제시해줘"라는 시스템 명령어를 조합하여 정형화된 JSON 분석 결과를 리턴받음.
  4. **결과 시각화**: 위험도 경고(빨강/노랑) 뱃지 부여 및 보완 가능한 권장 특약 복사 버튼 제공.

### 📋 기능 2: 셀프 서류 확인(근저당 등) 상세 가이드형 체크리스트
* **목적**: 앱에서 자동 조회할 수 없는 항목들을 사용자가 직접 대조 및 검증할 수 있도록 친절한 매뉴얼 및 링크 연동 지원.
* **구현 방안**:
  1. **가이드 아코디언 컴포넌트화**: 기존 자가진단 항목 하단에 접기/펴기 가능한 상세 매뉴얼 패널 추가.
  2. **구체적 실천 요령 탑재**:
     * *예: "을구 근저당 확인"* -> '대법원 인터넷등기소 접속 방법', '등기부 열람 수수료(700원)', '채권최고액 확인 및 시세의 70% 초과 여부 대조법' 등을 사진/텍스트 가이드로 수록.
     * *예: "중개사 자격 확인"* -> '국가공간정보포털(씨:real)' 중개업조회 바로가기 외부 링크 연동.

### ⚖️ 기능 3: 실시간 전세사기 판결문 RAG 학습 및 신종 수법 알림
* **목적**: 나날이 교묘해지는 최신 전세사기 법원 판결문을 RAG에 연동하여 신종 사기 패턴 분석 및 선제적 대응 조언.
* **구현 방안**:
  1. **판례 지식베이스 폴더 구축**: `SecondBrain` 내에 `대한민국 법률/판례/전세사기` 폴더를 신설하고 핵심 전세사기 판례(예: 신탁 사기, 동시 진행 바지집주인 사기 등) 마크다운 문서를 추가 적재.
  2. **RAG 엔진 확장**: `src/lib/rag.ts` 코드를 수정하여 법령 문서뿐만 아니라 판결문 마크다운 파일들도 함께 읽어들여 텍스트 덩어리(Chunk)로 인덱싱 및 임베딩하도록 범위 확장.
  3. **판결 기반 추론 프롬프트 고도화**: 챗봇 답변 시 **"최근 판결 판례에 따르면, 이와 유사한 수법(예: 임대인의 신탁회사 동의 없는 무단 임대)으로 보증금 반환이 거부된 사례가 있으니 주의하세요"**와 같은 판결 사례 중심의 위험 요인과 대응 방안을 리포트하게 함.

### 📖 기능 4: 계약서 전문 용어 사전 및 초고속 실시간 검색 (API 비용 0원 최적화)
* **목적**: 전세 계약서나 법률 조문에서 자주 등장하는 낯선 부동산/법률 전문 용어(예: 근저당권, 대항력, 신탁등기 등)의 개념을 즉각적으로 찾아볼 수 있는 지식 사전 제공.
* **구현 방안**:
  1. **정적 용어 데이터베이스 구축**: 부동산 필수 전문 용어 약 20여 개를 정의하여 프론트엔드 단독 데이터로 선언. (API 토큰 요금이 전혀 나가지 않는 100% 로컬 검색 방식)
  2. **실시간 검색 및 필터링 UI (`src/components/Glossary.tsx` 신설)**:
     * 사용자가 단어를 입력하면 실시간으로 결과를 매칭해 주는 인스턴트 검색 기능 제공.
     * 가나다 자음별 정렬을 지원하여 스크롤하며 사전식으로 읽어볼 수도 있도록 구현.
  3. **우측 탭 대시보드 통합**: 기존 우측의 [자가진단표]와 신설된 [용어 사전]을 부드럽게 토글 전환할 수 있도록 우측 탭 내비게이션 구축.

### 💰 기능 5: 국가 지원 주거 정책(대출, 청약, 지원금) 조회 기능 (API 비용 0원 최적화)
* **목적**: 전월세 세입자 및 무주택자들을 위해 정부에서 지원하는 맞춤형 대출 정책, 청약 요건, 주거 지원금 제도를 원스톱으로 확인 및 가이드 제공.
* **구현 방안**:
  1. **정부 공식 주거 복지 데이터셋 구축**:
     * **대출**: 청년/신혼부부 버팀목 전세대출, 디딤돌대출, 신생아 특례대출 등 조건 및 금리 정리.
     * **청약**: 공공분양 특별공급 요건, 청약저축 납입 조건 등 핵심 요약.
     * **지원금**: 청년 월세 한시 특별지원, 주거급여 등 대상 기준 명시.
  2. **사용자 맞춤형 필터링 UI (`src/components/SupportInfo.tsx` 신설)**:
     * 자신의 대상 속성(청년, 신혼부부, 일반무주택) 및 관심분야(대출, 청약, 지원금)를 클릭 시 맞춤형 제도가 즉시 로컬 필터링되는 스마트 셀렉터 탑재.
     * 공식 주택도시기금 및 LH 청약플러스 신청 페이지 바로가기 연동.
  3. **좌측 탭 대시보드 통합**: 기존 좌측 탭에 세 번째 메뉴로 [주거 정책 지원]을 추가하여 레이아웃을 일관되게 고수.

## 6. 비용 및 API 토큰 최적화 방안

API 호출 요금 및 속도 개선을 위해 프로토타입 단계부터 즉시 적용할 수 있는 비용 최적화 대책입니다.

### 💾 1. 임베딩 로컬 캐싱 (Embedding Caching)
* **문제점**: 서버를 재시작하거나 초기화할 때마다 전체 82개 법령 조항의 임베딩을 매번 OpenAI API로 다시 구하여 토큰이 중복 낭비됨.
* **해결책**: 최초 1회 생성된 임베딩 벡터 데이터를 로컬 파일(`src/lib/rag_cache.json`)로 파일 저장. 이후 부팅 시에는 OpenAI API를 호출하지 않고 로컬 파일에서만 벡터를 즉시 메모리에 올려 구동 속도를 단축하고 토큰 요금을 99% 이상 절약함.

### 🔍 2. RAG 컨텍스트 필터링 및 검색 Top-K 축소
* **문제점**: 질문을 할 때마다 유사도 점수가 낮아 관련성이 떨어지는 조문까지 LLM 프롬프트에 포함되어 토큰 요금이 증가함.
* **해결책**:
  * 유사도 검색 상위 개수를 기존 4개에서 2개(`topK: 2`)로 축소함.
  * 코사인 유사도 점수 임계값(Threshold, 예: 0.35 이상만 채택)을 걸어 두어, 관련 없는 쓰레기 조문이 LLM에 전송되지 않도록 필터링하여 프롬프트 길이를 절반 이하로 감소시킵니다.

### 💬 3. 대화 내역 슬라이딩 윈도우 (Sliding Window)
* **문제점**: 챗봇 대화가 길어질수록 누적된 전체 이전 대화 메시지가 통째로 전송되어 요금이 기하급수적으로 증가함.
* **해결책**: LLM 전송 시 최근 4~5개 메시지만 필터링하여 전송하는 슬라이딩 윈도우 기법을 적용하여 누적 요금을 제어함.

### 📄 4. 계약서 스캔 시 텍스트 필터링 및 로컬 OCR 검토
* **문제점**: 고해상도 이미지를 GPT-4o Vision API에 그대로 전송하면 높은 이미지 인풋 토큰 요금이 부과됨.
* **해결책**: 브라우저 로컬 단에서 가벼운 OCR 라이브러리(`Tesseract.js` 등)로 텍스트를 추출한 후, 불필요한 인적 사항이나 공백을 지우고 계약 조항 텍스트만 추출해 텍스트 기반 LLM 모델(`gpt-4o-mini`)에 태워 비용을 대폭 아낌.
